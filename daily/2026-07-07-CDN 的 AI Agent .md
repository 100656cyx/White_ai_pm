# 2026-07-07 Daily AI PM Log

## 1. 今天学习了什么

今天没有以概念学习为主，而是完成了一次 AI 产品实践：为 BytePlus CDN 的控制台设计 Agent 配置助手 PRD，并基于前端同学提供的原型工具生成可交互原型。

项目目标是用 AI 辅助完成从“需求”到“可交互原型”的转换，让产出符合现有设计规范，可直接用于产品评审和功能演示。可以参考线上的原型链接：https://veeixh.gf-preview.bytedance.net/

## 2. 今天动手实操了什么

### 用户与问题分析

在设计Agent助手前，我将目标用户分为三类，并分别分析 Agent 可以解决的问题：

- 新用户：对 CDN 和控制台中的抽象概念陌生，需要醒目入口和即时解释。
- 非技术用户：面对大量产品文档和复杂字段容易卡住，需要结合当前配置场景获得帮助。
- 熟练运维人员：更关注批量操作和配置效率，需要通过自然语言快速生成配置，并在确认后执行。

### 现状分析

我分析了火山引擎的 CDN Agent 配置体验，识别出以下问题：

- 入口较多，但定位和使用方式不统一。
- Agent 执行等待时间较长，过程缺少实时反馈。
- 用户离开当前对话后，任务可能中断。
- 只提供通用聊天入口，不能充分利用用户当前正在操作的配置上下文。

### Agent Friendly 交互设计

核心就是要：**Agent 得好用**

围绕“统一入口”和“场景内辅助”设计了两种入口：

- 主入口：在控制台导航中提供完整的 Agent Assistant 页面，承载历史会话、任务模板和配置记录。
- 场景入口：通过可移动、可隐藏的悬浮入口，在用户配置过程中随时唤起右侧对话框，并支持快捷键开启和关闭。

为了让 Agent 获得更准确的上下文，我设计了“Add to Agent for Assistance”交互：

- 用户可以把功能卡片或策略模板添加到对话中，快速询问功能含义和配置方法。
- 用户可以把具体配置字段添加到对话中，快速询问字段含义和推荐配置。
- 系统自动生成与当前对象相关的 Prompt，用户确认后再发起对话。

这个设计把“让用户重新描述上下文”改为“由界面把结构化上下文交给 Agent”，可以降低输入成本和表达歧义。

### 用户引导与反馈闭环

我设计了首次使用引导，依次介绍：

- Agent 主入口。
- 场景内唤起方式。
- 如何将功能、策略和字段加入对话。
- 如何使用预置 Prompt。

完成一次配置后，通过满意度卡片收集：

- 回答是否准确可靠。
- 功能是否容易使用。
- Agent 是否真正帮助用户完成工作。
- 用户的补充意见。

### 数据指标设计

我为后续效果评估设计了以下指标：

- Agent 入口曝光量和点击量。
- 会话开始、结束和异常中断次数。
- 配置草稿被应用、修改或放弃的情况。
- 使用漏斗：入口曝光 -> 点击 -> 会话开始 -> 任务模板使用 -> 配置草稿应用。
- Agent 响应时长分布和失败率。

## 3. 使用了哪些 AI 工具

- 工具：前端同学提供的 AI 原型生成工具。
- 使用场景：将产品需求转化为符合现有设计规范的可交互原型。
- 效果：快速生成可以用于评审和演示的交互页面，缩短从 PRD 到原型验证的时间。
- 不足：生成结果仍需要产品经理检查业务逻辑、交互一致性、文案准确性和异常状态，不能直接替代产品判断与设计评审。

## 4. 今天的关键收获

- Agent 产品设计不能只增加一个聊天框，关键是把业务场景和结构化上下文交给 Agent。
- 对不同熟练度的用户，Agent 的价值不同：新用户需要解释，普通用户需要辅助，专家用户需要提效。
- 配置类 Agent 需要明确的确认环节，避免直接执行高风险操作。
- Agent 的过程反馈、任务持续性和失败恢复能力，会直接影响用户是否信任它。
- **AI 生成原型提高了表达和验证效率，但产品经理仍要对需求逻辑和最终体验负责。**


## 5. 对 AI 产品经理岗位的新理解

这次实践让我开始从 Agent Friendly 的角度审视传统控制台：

- 页面中的功能、策略、字段应该可以被结构化引用，而不是只让 Agent 理解用户的自由文本。
- Agent 应该知道用户当前所在页面、选中的对象和准备完成的动作。
- 生成配置后需要提供清晰的预览、确认和修改机制。
- 产品指标不能只看聊天次数，还要看配置是否被采用、任务是否完成以及失败发生在哪一步。

AI 产品经理需要同时理解业务流程、Agent 能力边界、交互控制点和效果评估方法。
