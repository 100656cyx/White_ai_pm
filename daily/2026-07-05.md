## 2. ReAct 运行模式

ReAct 的基本流程是：

1. 用户提交任务。
2. Agent 先进行 `thought`，判断下一步应该做什么。
3. 如果需要外部能力，就通过 `action` 调用工具。
4. 工具返回结果后，Agent 通过 `observation` 分析执行结果。
5. 重复 `thought -> action -> observation`，直到可以输出 `final_answer`。

示例结构：

```text
<task>告诉我埃菲尔铁塔有多高？</task>
<thought>我需要找到埃菲尔铁塔的高度。可以使用搜索工具。</thought>
<action>get_height("埃菲尔铁塔")</action>
<observation>埃菲尔铁塔的高度约为330米（包含天线）。</observation>
<thought>搜索结果显示了高度。我已经得到答案了。</thought>
<final_answer>埃菲尔铁塔的高度约为330米。</final_answer>
```

今天理解到：ReAct 这种运行模式很大程度上是通过系统提示词约束模型输出格式，再由 Agent 主程序解析模型输出、决定是否调用工具来实现的。

## 3. 如何通过代码实现 ReAct

结合本地 `agent.py` 示例代码，今天理解到 ReAct Agent 可以拆成几个部分：

- Agent 类：封装整体流程，例如 `ReActAgent`。
- LLM 调用：负责把系统提示词、用户问题、历史 observation 发送给模型。
- 工具列表：把外部动作注册成函数，例如读文件、写文件、执行终端命令。
- 系统提示词：告诉模型必须按 `thought/action/observation/final_answer` 格式工作。
- 主循环：不断请求模型、解析 action、调用工具、追加 observation，直到得到 final answer。

因此，Agent 可以理解为：

- 模型
- 工具，也就是可被调用的函数
- 工具调用声明和约束
- Agent 框架主程序，例如 ReAct 主循环

## 4. 关于工具调用的新理解

今天一个关键认知是：

大模型本身不会真的调用工具。模型只是根据上下文生成“我想调用某个工具”的文本或结构化请求。真正执行工具的是 Agent 框架里的工具调用组件。

这意味着：

- 工具权限控制应该放在 Agent 框架层，而不是完全交给模型。
- 高风险工具必须有人类确认，例如执行终端命令、删除文件、发布配置。
- 工具返回结果需要重新进入上下文，让模型继续推理。

## 5. ReAct 的局限

我认为 ReAct 的优点是流程清晰，适合解释 Agent 如何一步步完成任务。

但它也有局限：

- 每一步都基于当前局部判断，容易缺少整体规划。
- 在复杂任务中，可能陷入局部复杂解。
- 如果 observation 太多，容易增加上下文负担。
- 对需要长期目标拆解的任务，可能不如先规划再执行稳定。

## 6. Plan and Execute 模式

今天还学习到另一种运行模式：Plan and Execute。

这个模式不是边想边做，而是先生成整体计划，再逐步执行，并在执行过程中支持重新规划。

它通常包括：

- Plan 模型：先整理任务计划。
- Re-Plan 模型：根据执行结果调整计划。
- 执行 Agent：负责具体动作执行，可以是 ReAct Agent。
- Agent 主程序：负责调度计划、执行和重新规划。

我的理解是：Plan and Execute 更适合复杂任务，ReAct 更适合理解和实现基础 Agent 循环。

## 8. 对 AI 产品经理岗位的新理解

- AI 产品经理需要理解“模型不会直接行动，Agent 框架才负责执行动作”这个边界。
- 评估 Agent 产品时，不能只看模型能力，还要看工具设计、权限控制、执行链路和失败处理。
- ReAct、Plan and Execute 这类运行模式，会影响产品体验、执行稳定性和可解释性。
- 面向开发者的 Agent 产品，要把工具调用过程设计得清晰、可观察、可中断。
