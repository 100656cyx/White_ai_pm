# 2026-07-09 Daily AI PM Log

## 1. 今天学习了什么

主题：OpenClaw（1）。今天重点理解 OpenClaw 这类 Agent 运行平台，和之前学习的 ReAct、Plan and Execute 之间的关系。

参考资料：

- https://github.com/openclaw/openclaw
- https://docs.openclaw.ai/concepts/architecture

核心理解：

ReAct 解决的是“Agent 如何循环思考和调用工具”。OpenClaw 这类工程化 Agent 系统解决的是：如何让这个循环长期、稳定、安全地运行，并能够被真实用户在真实场景中使用。

也就是说，Agent 的底层运行逻辑仍然可以与 ReAct、Plan and Execute 类似，但真正产品化落地时，还需要补齐会话持久化、消息渠道、定时任务、权限、沙箱、插件和多 Agent 等工程能力。

## 2. OpenClaw 的任务运行流程

我今天对 OpenClaw 的任务链路理解如下：

```text
应用场景：Lark / Telegram / CLI / Web
        |
      Gateway
        |
Session / Context / Memory
        |
    Agent Loop
        |
LLM 推理并生成 Tool Call
        |
OpenClaw 执行工具
        |
文件 / Shell / 浏览器 / API 等真实动作
```

一次任务大致会经历：

1. 接收用户输入的 prompt。
2. 找到对应会话，加载历史上下文、相关 skill 和 memory。
3. 调用模型，让模型判断下一步动作。
4. 模型输出 Tool Call 请求和参数。
5. OpenClaw 执行工具，并把执行结果返回给模型。
6. 模型继续判断是否完成任务，如果未完成则继续循环。

这个过程和 ReAct 的“思考 -> 行动 -> 观察 -> 再思考”一致，但 OpenClaw 关注的是把这个循环接到真实系统里，并控制它的可靠性和风险。

## 3. 工程化能力理解

### 会话持久化：Session

最基础的 ReAct 程序通常是：

```text
接收用户问题 -> 发给模型 -> Agent 执行工具 -> 循环 -> 程序结束
```

程序结束后，执行过程通常不会被保存，上下文也不可追溯。

OpenClaw 会保存：

- 用户和 Agent 的历史消息。
- Tool Call 的执行记录和结果。
- 当前任务状态。
- 使用的模型、消息来源和运行轨迹。

产品价值：

- 用户可以随时继续之前的对话和任务。
- 长任务可以被拆分、暂停、恢复。
- 不同任务可以区分隔离。
- 发生中断后，可以根据历史状态恢复或定位问题。

我的理解是：Session 是 Agent 产品化的基础设施。如果没有持久化，会话只能像一次性脚本，无法成为真正的工作助手。

### 长期记忆：Memory

今天明确区分了“会话记录”和“长期记忆”：

- Session：记录某一次具体任务中的消息、工具调用和状态。
- Memory：记录用户长期稳定的偏好、事实和工作方式。

例如：

```text
MEMORY.md          # 长期稳定的信息
memory/2026-07-09.md  # 当天详细记录
```

产品价值：

- Agent 可以保持长期风格和目标一致性。
- 不同任务之间可以复用稳定偏好，而不是每次都重新说明。
- 记忆需要被写入、更新、过期和删除，不能无限增长。

我的理解是：Memory 不是越多越好，而是要在当前任务中“恰好有用”。如果记忆过期、错误或过度暴露，反而会影响 Agent 判断。

### 消息渠道：Channels

普通 ReAct 程序往往只接收命令行输入。OpenClaw 可以连接多种客户端和消息平台，例如 Lark、Telegram、WhatsApp、Discord、Web、CLI 等。

消息渠道的基本流程是：

```text
接收平台消息
-> 识别用户和会话
-> 转换成统一消息格式
-> 交给 Agent 处理
-> 把结果发回原渠道
```

产品价值：

- 用户不需要打开一个新产品，可以在日常工作工具里直接调用 Agent。
- Agent 从“一个工具”变成“一个随处可用的工作入口”。

需要注意的是，不同渠道的消息长度、文件格式、权限、群聊/私聊关系和流式输出能力都不同，因此 Channels 不只是接入入口，也涉及体验适配。

### 定时任务：Cron

Agent 的触发方式不一定只能来自用户主动输入，也可以由时间触发。

例如我自己的学习仓库工作流：

```text
每天 22:30 -> 检查当天是否有学习记录 -> 如果有内容且未提前上传 -> 总结并推送到 GitHub
```

产品价值：

- Agent 可以承担周期性、重复性的自动化任务。
- 用户不需要每次主动唤起。
- 适合日报、巡检、同步、监控和提醒等场景。

但 Cron 任务需要处理重复执行、幂等、失败重试、超时和通知，否则自动化会带来新的不确定性。

### 权限控制：Permission

模型返回 Tool Call，并不代表本地系统应该直接执行。

不同动作的风险等级不同：

- 读取文件：风险较低。
- 修改项目文件：中等风险。
- 推送 GitHub：中高风险。
- 删除文件、部署生产应用：高风险。

OpenClaw 支持类似以下权限模式：

- `deny`：完全禁止。
- `allowlist`：只允许白名单命令。
- `ask`：不在白名单时询问用户。
- `auto`：由安全策略评估，必要时询问用户。
- `full`：无需确认，直接执行。

产品理解：

权限控制的核心是 Human in the Agent Loop。Agent 可以提出动作，但高风险动作需要用户预览、确认或禁止。

适合的分级方式是：

- 只读查询：可以自动执行。
- 生成草稿：可以自动执行。
- 修改数据：需要预览。
- 外部发布：需要确认。
- 删除或生产操作：强确认或禁止。

### 沙箱：Sandbox

权限控制解决的是“能不能做”，沙箱解决的是“在哪里做、影响范围有多大”。

沙箱可以限制：

- Agent 能看到哪些目录。
- 工作区只读还是可写。
- 是否允许访问宿主机。
- 是否可以访问网络。
- 不同 Agent 是否共享环境。
- 是否使用 Docker 等方式隔离。

典型模式可以理解为：

- `none`：看不到真实工作区。
- `ro`：只能读取指定工作区。
- `rw`：可以读写指定工作区。

产品理解：

权限控制、沙箱和用户确认是三层不同保护，不能相互替代。

例如，如果只是让 Agent 辅助写邮件，就不需要赋予它删除本地项目文件的能力。如果是 vibe coding 一个项目，依赖和文件变更也应该尽量限制在当前项目目录内。

### 插件：Plugin

插件机制让 Agent 的能力可以扩展，而不是全部写死在主程序中。

插件可能扩展：

- Tools：新增可调用工具。
- Model Provider：接入不同模型。
- Channel：接入新的消息平台。
- Hook：在工具执行前后增加检查或记录。
- Memory：替换或增强记忆实现。
- CLI Command：新增命令行能力。

产品价值：

- 平台能力可以被第三方或内部团队扩展。
- 不同业务场景可以复用同一个 Agent 底座。
- 权限、配置和工具说明可以被标准化管理。

我的理解是：插件机制是 Agent 平台生态化的前提。没有插件，Agent 只能是一个固定应用；有了插件，才可能变成可扩展平台。

### 多 Agent：Multi-Agent

复杂任务可以由主 Agent 分配给多个子 Agent。

例如：

- 搜索 Agent：负责资料收集。
- 整理 Agent：负责结构化总结。
- 代码 Agent：负责实现和验证。
- 安全 Agent：负责检查风险动作。

产品价值：

- 降低单个 Agent 的上下文压力。
- 让不同 Agent 拥有不同工具和权限。
- 将复杂任务拆成更清晰的职责边界。

风险是：多 Agent 会带来协调成本、上下文传递损耗和权限边界复杂化，因此不应该为了“看起来智能”而滥用。

## 4. 和 ReAct 的关系

今天形成的判断是：

ReAct 是 Agent 的基础运行模式之一，像一个工具调用的 `while` 循环：

```text
思考 -> 调用工具 -> 观察结果 -> 继续思考 -> 直到任务完成
```

OpenClaw 不是替代 ReAct，而是把 ReAct 这类运行模式放进一个更完整的工程系统里。

可以这样理解：

| 模块 | 解决的问题 |
| --- | --- |
| ReAct | Agent 如何边思考边行动 |
| Session | 任务过程如何保存和恢复 |
| Memory | 长期偏好和事实如何复用 |
| Channel | 用户从哪里与 Agent 交互 |
| Cron | 没有用户输入时如何定时触发 |
| Permission | 哪些动作可以执行 |
| Sandbox | 动作只能影响哪些环境 |
| Plugin | Agent 能力如何扩展 |
| Multi-Agent | 复杂任务如何分工 |

## 5. 对 AI 产品经理岗位的新理解

今天最大的收获是：Agent 产品经理不能只理解模型和 Prompt，还要理解 Agent 系统为什么需要工程化能力。

一个能产品化的 Agent，需要同时回答：

- 如何接收真实用户输入？
- 如何记住上下文和长期偏好？
- 如何把任务拆解给模型和工具？
- 如何控制工具调用风险？
- 如何让任务失败后可恢复？
- 如何让用户知道 Agent 做了什么？
- 如何接入不同业务系统和工作渠道？

从产品角度看，模型能力决定 Agent 的上限，但工程系统决定它能不能被稳定、安全、长期地使用。
